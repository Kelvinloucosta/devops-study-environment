Homebrew install first

Homebrew is the package manager for macOS — think of it as an App Store for developer tools, but from the terminal. Every other tool in this guide installs and uninstalls through Homebrew. Install this before anything else.

Run this in Terminal

/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

After install — add Homebrew to your PATH

# The installer will show you this command — run it:
echo 'eval "$(/opt/homebrew/bin/brew shellenv)"' >> ~/.zshrc
source ~/.zshrc

Verify it works

brew --version
You mostly won't use Homebrew directly — it runs in the background when you install other tools. But here are the commands you'll occasionally need:
1
Install any tool
Add --cask for apps with a GUI (like Docker Desktop or UTM). Without it, for CLI tools.
brew install <tool>        # CLI tools
brew install --cask <app>  # GUI apps
2
Update everything
Fetches the latest versions of all installed tools.
brew update && brew upgrade
3
See what's installed
brew list
Remove Homebrew last, after uninstalling everything else through it.

Remove Homebrew completely

/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/uninstall.sh)"

Also remove from your shell config

# Open ~/.zshrc and delete this line:
# eval "$(/opt/homebrew/bin/brew shellenv)"
nano ~/.zshrc

Docker pick one

Docker lets you run isolated Linux containers on your Mac — like lightweight virtual machines that start in seconds. You use them to run apps, practice with databases, simulate servers, and test deployments without affecting your Mac. Pick one option below: Colima (free, terminal-based) or Docker Desktop (free for personal use, has a GUI).

Colima recommended
Terminal-based, lightweight, 100% free forever. No GUI, no account needed.
Docker Desktop
Has a visual dashboard to see containers. Free for personal/study use. Requires a free account.

Install Colima + Docker CLI

brew install colima docker docker-compose
That's it for installation. You start Colima when you want to use Docker (see the Use tab).

Install Docker Desktop

brew install --cask docker
After install, open Docker from Applications and sign in with a free Docker account. The daemon starts automatically.

Verify Docker is working (both options)

docker version
Think of Docker like this: an image is a recipe (e.g. "Ubuntu Linux"), and a container is a running instance of that recipe. You can spin up and throw away containers freely — they don't affect your Mac at all.
1
Start Docker (Colima users only)
Run this before using any Docker command. Docker Desktop starts automatically when you open the app.
colima start
2
Run a Linux container interactively
This drops you into a real Ubuntu shell. You can install packages, run commands — all isolated. Type exit to leave.
docker run -it ubuntu bash
3
See running containers
docker ps
4
Stop a container
docker stop <container-name>
5
Build your own image from a Dockerfile
docker build -t myapp .
docker run myapp
6
Run multiple services together (compose)
Useful for app + database setups. Put a compose.yaml in your project folder, then:
docker compose up -d    # start in background
docker compose down     # stop and remove
7
Stop Docker when done studying (Colima)
Frees up RAM immediately. Your containers are saved and resume next time.
colima stop
Option A — Remove Colima
Uninstalls Colima and the Docker CLI tools.
brew uninstall colima docker docker-compose

Remove leftover data (images, containers)

rm -rf ~/.colima
Option B — Remove Docker Desktop
brew uninstall --cask docker

Remove leftover data

rm -rf ~/Library/Group\ Containers/group.com.docker
rm -rf ~/Library/Containers/com.docker.docker
Removing Docker also removes all images and containers. This can free up several GB of disk space.

UTM free

UTM lets you run full Linux virtual machines (VMs) on your Mac — like having a real separate computer running Ubuntu or Debian. Use it when Docker containers aren't enough: for practicing with systemd services, setting up SSH servers, networking configs, or anything that needs a complete OS.

Install UTM

brew install --cask utm

Get a Linux image — use the built-in Gallery (easiest)

Open UTM → click Browse UTM Gallery → pick Debian or Ubuntu → click Open in UTM.

No .iso download needed. The VM will appear in the sidebar automatically.
Default credentials for the Gallery VMs:
User: debian  ·  Password: debian
Root: root  ·  Password: password

⚠️ If you picked Debian 10 (Buster) — fix the repositories first

Debian 10 reached end-of-life in 2022. Its package repositories were archived, so apt-get update will fail with 404 errors out of the box. Run the commands below once to fix it before installing anything. 
# Point apt to the Debian archive mirror
sudo bash -c 'cat > /etc/apt/sources.list << EOF
deb http://archive.debian.org/debian buster main contrib non-free
deb http://archive.debian.org/debian-security buster/updates main
EOF'

# Now update and install normally
sudo apt-get update
sudo apt-get install -y curl
If you want a more up-to-date system, grab Debian 12 (Bookworm) or Ubuntu 24.04 from the Gallery instead — both are supported until 2027+.
Think of UTM VMs like a separate computer inside your Mac. It boots Linux, you use it, then suspend or shut it down. Your Mac is unaffected.

The most comfortable way to use the VM is via SSH from your Mac terminal — no need to type inside the UTM window. Follow the steps below to set that up.
1
Before starting — change the network mode
The default network (Emulated VLAN) doesn't allow SSH from your Mac. You need to switch it to Shared Network first. Do this once, before starting the VM.
Stop the VM if it's running → click the edit/settings icon in UTM → go to Network → change Network Mode from Emulated VLAN to Shared Network → save → start the VM again.
2
Start the VM and log in inside the UTM window
Click ▶ to start, then log in directly in the UTM window to find the IP.
login: debian
password: debian
3
Find the VM's IP address
Run this inside the UTM window. Look for the inet line — it'll be something like 192.168.64.x.
ip addr show | grep inet
4
SSH from your Mac terminal
Now open your Mac terminal and connect. Replace 192.168.64.x with the IP you found above.
ssh debian@192.168.64.x
# Password: debian
You're now inside the VM from your Mac terminal. No need to use the UTM window anymore.
5
Use it like a real Linux server
Install packages, configure services, practice anything you'd do on a remote server.
sudo apt update && sudo apt install nginx
systemctl status nginx
6
Suspend when done (saves the VM state)
In UTM: Machine → Suspend. Saves everything instantly and frees RAM. Resumes in seconds next time — no need to find the IP again.
7
Or shut down fully from inside the VM
sudo shutdown now

Remove the UTM app

brew uninstall --cask utm

Remove VM data (this is where disk space lives)

rm -rf ~/Library/Containers/com.utmapp.UTM
rm -rf ~/Documents/UTM
Warning: this permanently deletes all your VMs and their data. Make sure you don't need them anymore.

mise free · open source

mise manages versions of DevOps tools like kubectl, terraform, and helm — per project folder. This means your study folder can use kubectl v1.29 while your work folder uses a different version. They never conflict.

Install mise

brew install mise

Activate it in your shell

# Add to ~/.zshrc so it loads on every terminal session
echo 'eval "$(mise activate zsh)"' >> ~/.zshrc
source ~/.zshrc

Verify

mise --version
How it works: when you run mise use kubectl@1.29 inside a folder, mise creates a .mise.toml file there. Every time you enter that folder, that version of kubectl becomes active. Leave the folder — it goes back to whatever version your work setup uses.
1
Go to your study folder and pin tool versions
cd ~/studies/k8s
mise use kubectl@1.29
mise use helm@3.14
mise use terraform@1.7
This creates a .mise.toml file. Commit it to git so teammates get the same versions automatically.
2
Install the pinned tools (in a fresh clone)
mise install
3
Check which version is active right now
mise list
kubectl version --client
4
Clean up unused versions
mise prune

Uninstall mise

brew uninstall mise

Remove all downloaded tool versions and config

rm -rf ~/.local/share/mise
rm -rf ~/.config/mise

Remove from shell config

# Open ~/.zshrc and delete this line:
# eval "$(mise activate zsh)"
nano ~/.zshrc

direnv free · open source

direnv automatically loads and unloads environment variables when you enter or leave a folder. For example: your study folder loads KUBECONFIG=~/.kube/config-lab automatically when you cd into it — and removes it when you leave. Your work cluster config is never accidentally touched.

Install direnv

brew install direnv

Hook it into your shell

echo 'eval "$(direnv hook zsh)"' >> ~/.zshrc
source ~/.zshrc
How it works: you create a file called .envrc in your project folder with the env vars you want. direnv loads them when you enter the folder and removes them when you leave. No manual export/unset needed.
1
Create a .envrc in your study folder
# Example for a k8s study folder
cat > ~/studies/k8s/.envrc << 'EOF'
export KUBECONFIG=~/.kube/config-lab
export AWS_PROFILE=lab
export ENVIRONMENT=dev
EOF
2
Allow direnv to load it (one-time per file)
For security, direnv asks you to approve each .envrc before loading it.
cd ~/studies/k8s
direnv allow
3
Watch the magic happen
# Enter the folder — vars load automatically
cd ~/studies/k8s
direnv: loading ~/studies/k8s/.envrc
direnv: export +KUBECONFIG +AWS_PROFILE +ENVIRONMENT

# Leave the folder — vars unload automatically
cd ~
direnv: unloading
4
Edit the .envrc later
direnv edit .

Uninstall direnv

brew uninstall direnv

Remove config

rm -rf ~/.config/direnv

Remove from shell config

# Open ~/.zshrc and delete this line:
# eval "$(direnv hook zsh)"
nano ~/.zshrc

act free · open source

act runs your GitHub Actions workflows locally using Docker — without pushing to GitHub. Perfect for learning CI/CD: you write a workflow, test it instantly on your Mac, fix it, and only push when it's working. No wasted build minutes, no waiting for GitHub runners.

Install act

brew install act
act requires Docker to run. Make sure you have Colima running (colima start) or Docker Desktop open before using act.
How it works: act reads the .github/workflows/ folder in your repo and runs the workflows inside Docker containers — exactly like GitHub's servers would, but on your Mac.
1
Go to a repo that has GitHub Actions workflows
cd ~/studies/github-actions/my-repo
2
List all available workflows
act -l
3
Run the workflow that triggers on push
First run downloads a Linux Docker image (~500MB). Subsequent runs are instant.
act push
4
Run only a specific job
act -j build
act -j test
5
Preview what would run without actually running it
act --dry-run

Uninstall act

brew uninstall act

Remove Docker images downloaded by act

# List act-related images
docker images | grep act

# Remove them
docker images | grep act | awk '{print $3}' | xargs docker rmi
